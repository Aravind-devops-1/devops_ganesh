# Kubernetes Secrets – Complete Hands-On Guide

## 1. What Are Kubernetes Secrets?

* **Secrets** are Kubernetes objects used to store **sensitive information**, such as:

  * Passwords
  * API keys
  * Tokens
  * Certificates

* They are **safer than ConfigMaps** because Kubernetes can store them in base64 and integrate with RBAC to limit access.

**Analogy:**

* Think of ConfigMaps as a whiteboard for public instructions, while Secrets are a **locked drawer with sensitive documents**.

---

## 2. Why Use Secrets?

* Keeps sensitive data **out of source code**
* Prevents accidental exposure
* Allows safe injection into Pods
* Can be used for environment variables, volumes, and image pull credentials

---

## 3. Types of Secrets

1. **Opaque (Default)** – Stores arbitrary key–value pairs (most common).
2. **docker-registry** – Stores Docker registry credentials for pulling private images.
3. **tls** – Stores TLS certificates (key + cert) for HTTPS.
4. **Basic Auth / SSH Keys** – Store authentication information for apps or Git.

---

## 4. Hands-On: Creating and Using Secrets

### 4.1. Opaque Secret (Simple Key–Value)

#### Secret YAML – `app-secret.yaml`

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  DB_USER: YWRtaW4=        # 'admin' in base64
  DB_PASS: cGFzc3dvcmQ=    # 'password' in base64
```

**Commands to Apply & View**

```bash
kubectl apply -f app-secret.yaml
kubectl get secret app-secret
kubectl describe secret app-secret
kubectl get secret app-secret -o yaml   # Shows base64 encoded data
```

**Explanation:**

* `DB_USER` and `DB_PASS` are stored securely.
* Base64 encoding is **not encryption**, but avoids plain text.

---

### 4.2. Inject Secret as Environment Variables

#### Pod YAML – `pod-env-secret.yaml`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-env-pod
spec:
  containers:
  - name: app
    image: nginx
    env:
    - name: DB_USER
      valueFrom:
        secretKeyRef:
          name: app-secret
          key: DB_USER
    - name: DB_PASS
      valueFrom:
        secretKeyRef:
          name: app-secret
          key: DB_PASS
```

**Explanation:**

* Each key in the secret becomes an environment variable.
* The application reads DB credentials securely from the environment.

---

### 4.3. Inject Secret as Volume (File-Based)

#### Pod YAML – `pod-volume-secret.yaml`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-volume-pod
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: secret-vol
      mountPath: "/etc/secrets"
      readOnly: true
  volumes:
  - name: secret-vol
    secret:
      secretName: app-secret
```

**Explanation:**

* Secret keys appear as files in `/etc/secrets`.
* Useful for apps that read config from files.

---

### 4.4. TLS Secret

#### Secret YAML – `tls-secret.yaml`

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: tls-secret
type: kubernetes.io/tls
data:
  tls.crt: <base64-encoded-cert>
  tls.key: <base64-encoded-key>
```

**Usage:**

* For Ingress or Pods that require HTTPS.
* Mounted as volume or referenced in Ingress TLS section.

---

### 4.5. Docker Registry Secret

#### Secret YAML – `docker-secret.yaml`

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: reg-secret
type: kubernetes.io/dockerconfigjson
data:
  .dockerconfigjson: <base64-encoded-docker-config>
```

**Usage:**

* Allows pulling images from **private Docker registries**.
* Reference in Pod `imagePullSecrets`:

```yaml
imagePullSecrets:
- name: reg-secret
```

---

### 4.6. Optional Secret Reference

#### Pod YAML – `pod-optional-secret.yaml`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: optional-secret-pod
spec:
  containers:
  - name: app
    image: nginx
    envFrom:
    - secretRef:
        name: non-existing-secret
        optional: true
```

**Explanation:**

* Pod will start **even if the Secret does not exist**.
* Useful for optional configurations.

## 5. Commands Cheat Sheet

```bash
# Create secret from YAML
kubectl apply -f app-secret.yaml

# Create secret from literal values
kubectl create secret generic app-secret-literal --from-literal=DB_USER=admin --from-literal=DB_PASS=password

# Create secret from file
kubectl create secret generic app-secret-file --from-file=app.conf

# View secrets
kubectl get secret
kubectl describe secret app-secret

# Decode secret
kubectl get secret app-secret -o jsonpath="{.data.DB_USER}" | base64 --decode
```

---

## 6. Important Best Practices

* **Do not store plain-text secrets in Git**.
* Use **RBAC** to control who can read Secrets.
* For production, consider **external secret management** (Vault, AWS Secrets Manager, Azure Key Vault).
* Avoid large Secrets (keep <1 MiB).
* Rotate Secrets periodically.
