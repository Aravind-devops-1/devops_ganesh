# Hands-on: Kubernetes Volume Types

## 1. emptyDir Volume – Hands-on

### Goal

Understand **temporary, pod-scoped storage**.

---

### YAML: Pod with emptyDir

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
  - name: app
    image: busybox
    command: ["sh", "-c", "echo Hello > /data/file.txt && sleep 3600"]
    volumeMounts:
    - name: temp-vol
      mountPath: /data
  volumes:
  - name: temp-vol
    emptyDir: {}
```

---

### Commands

```bash
kubectl apply -f emptydir.yaml
kubectl exec -it emptydir-demo -- cat /data/file.txt
kubectl delete pod emptydir-demo
```

---

### Observations

* File exists while Pod is running
* After Pod deletion → data is lost
* Storage is node-local
* Suitable only for temporary data

---

## 2. PersistentVolume (PV) – Hands-on

### Goal

Create **actual storage resource** in the cluster.

---

### YAML: PersistentVolume (hostPath)

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-demo
spec:
  capacity:
    storage: 1Gi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: /mnt/pv-data
```

---

### Commands

```bash
kubectl apply -f pv.yaml
kubectl get pv
kubectl describe pv pv-demo
```

---

### Observations

* PV status shows `Available`
* Not usable by Pods directly
* Exists independent of Pods

---

## 3. PersistentVolumeClaim (PVC) – Hands-on

### Goal

Request storage from available PVs.

---

### YAML: PVC

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-demo
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
```

---

### Commands

```bash
kubectl apply -f pvc.yaml
kubectl get pvc
kubectl get pv
```

---

### Observations

* PVC status becomes `Bound`
* PV status changes from `Available` → `Bound`
* PVC and PV are now linked

---

## 4. Pod Using PVC – Hands-on

### Goal

Persist data beyond Pod lifecycle.

---

### YAML: Pod with PVC

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pvc-pod-demo
spec:
  containers:
  - name: app
    image: busybox
    command: ["sh", "-c", "echo PersistentData > /data/info.txt && sleep 3600"]
    volumeMounts:
    - name: storage
      mountPath: /data
  volumes:
  - name: storage
    persistentVolumeClaim:
      claimName: pvc-demo
```

---

### Commands

```bash
kubectl apply -f pvc-pod.yaml
kubectl exec -it pvc-pod-demo -- cat /data/info.txt
kubectl delete pod pvc-pod-demo
kubectl apply -f pvc-pod.yaml
kubectl exec -it pvc-pod-demo -- cat /data/info.txt
```

---

### Observations

* Data **persists even after Pod deletion**
* Confirms persistent storage behavior
* PVC reuses the same PV

---

## 5. emptyDir with Memory (RAM-backed) – Hands-on

### Goal

See **RAM-based temporary storage**.

---

### YAML: emptyDir with Memory

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-memory
spec:
  containers:
  - name: app
    image: busybox
    command: ["sh", "-c", "mount | grep data && sleep 3600"]
    volumeMounts:
    - name: mem-vol
      mountPath: /data
  volumes:
  - name: mem-vol
    emptyDir:
      medium: Memory
```

---

### Commands

```bash
kubectl apply -f emptydir-memory.yaml
kubectl exec -it emptydir-memory -- mount | grep data
```

---

### Observations

* Volume is mounted as `tmpfs`
* Uses node RAM
* Faster I/O
* Data lost on Pod deletion or node reboot

---

## 6. StorageClass (Basic Demo)

### Goal

Understand **dynamic provisioning concept** (even if not fully functional on all setups).

---

### YAML: StorageClass

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: manual
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
```

---

### Commands

```bash
kubectl apply -f storageclass.yaml
kubectl get storageclass
kubectl describe storageclass manual
```

---

### Observations

* StorageClass defines *how* storage is provisioned
* Used by PVCs for dynamic PV creation
* No data stored by StorageClass itself

---

# Final Summary – Hands-on Learnings

| Volume Type       | Persistent | Scope     | Key Observation           |
| ----------------- | ---------- | --------- | ------------------------- |
| emptyDir          | No         | Pod       | Data lost on Pod deletion |
| emptyDir (Memory) | No         | Pod       | Uses node RAM (tmpfs)     |
| PV                | Yes        | Cluster   | Actual storage resource   |
| PVC               | Yes        | Namespace | Storage request           |
| Pod + PVC         | Yes        | App-level | Data survives Pod restart |
| StorageClass      | Yes        | Cluster   | Enables automation        |

---

## Interview Memory Line

> emptyDir is temporary and pod-scoped.

> PV is the actual storage.

> PVC is the request.

> StorageClass automates provisioning.
