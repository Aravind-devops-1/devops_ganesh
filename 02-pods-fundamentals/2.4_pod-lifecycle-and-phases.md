# Pod Lifecycle and Phases

## What Is Pod Lifecycle?

The **Pod lifecycle** describes the different **states a Pod goes through** from creation to termination.

A Pod is **not permanent**.
It is created, runs, and eventually ends.

## Pod Phases (High Level)

Kubernetes represents a Pod’s lifecycle using **phases**.

```
Pending → Running → Succeeded => Failed / Unknown
```

## Pod Phases Explained

### 1. Pending

* Pod is accepted by Kubernetes
* Containers are **not running yet**
* Image may still be downloading
* Pod may be waiting for scheduling

Common reasons:

* Image pull delay
* No available node resources

Command:

```bash
kubectl get pods
```


### 2. Running

* Pod is scheduled to a node
* At least one container is running
* Application is executing

Check details:

```bash
kubectl describe pod <pod-name>
```

### 3. Succeeded

* All containers completed successfully
* Common for **Jobs** and **batch workloads**

Example:

* Backup job
* Data processing task


### 4. Failed

* One or more containers exited with error
* Pod will not recover automatically (unless controlled by a controller)

Check logs:

```bash
kubectl logs <pod-name>
```


### 5. Unknown

* Kubernetes cannot determine Pod state
* Usually due to node communication issues


## Important Notes

* Pod **phase is not container status**
* A Pod can be `Running` even if:

  * One container crashed and restarted
* For production workloads:

  * Pods are managed by **Deployments**, not directly


## Pod Restart Behavior (Quick Intro)

Restart behavior depends on **restartPolicy**:

* `Always` (default)
* `OnFailure`
* `Never`

(This is covered in detail later)


## Useful Commands for Pod Lifecycle

```bash
kubectl get pods
kubectl get pods -o wide
kubectl describe pod <pod-name>
kubectl logs <pod-name>
kubectl delete pod <pod-name>
```


## Key Takeaways

* Pod lifecycle is phase-based
* Pods are temporary
* Phases show **high-level state**, not details
* Controllers handle recovery, not Pods
