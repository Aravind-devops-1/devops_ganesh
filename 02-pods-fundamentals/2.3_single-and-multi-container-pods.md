# Single-Container and Multi-Container Pods

## 1. What Is a Single-Container Pod?

A **single-container Pod** runs **one container** inside it.

This is:

* The **most common**
* The **recommended default**
* Easier to manage and debug

### When to Use

* Web applications
* APIs
* Microservices
* Background workers

### Simple Example

One Pod → One container → One responsibility


## 2. What Is a Multi-Container Pod?

A **multi-container Pod** runs **more than one container** inside the same Pod.

All containers:

* Share the same IP
* Share the same volumes
* Start and stop together


## 3. Why Multi-Container Pods Exist

Multi-container Pods are used when containers must:

* Work together closely
* Share data or network
* Have the same lifecycle

They are **not** used for unrelated services.



## 4. Common Multi-Container Pod Patterns

| Pattern Name           | Containers Involved                                          | Purpose                                                       | Example                 |
| ---------------------- | ------------------------------------------------------------ | ------------------------------------------------------------- | ----------------------- |
| **Sidecar Pattern**    | Main app container<br>Helper container (logging, monitoring) | Extends or supports the main application without modifying it | App + log collector     |
| **Ambassador Pattern** | App container<br>Proxy container                             | Acts as a proxy to handle external communication for the app  | App + API gateway       |
| **Adapter Pattern**    | App container<br>Data transformer container                  | Transforms or normalizes data to a required format            | App + metrics formatter |


## 5. Single vs Multi-Container Pods (Quick View)

| Aspect      | Single-Container | Multi-Container  |
| ----------- | ---------------- | ---------------- |
| Complexity  | Low              | Medium           |
| Use case    | Most workloads   | Special cases    |
| Debugging   | Easy             | Slightly complex |
| Recommended | Yes              | Only when needed |


## 6. Best Practices (Very Important)

* Prefer **single-container Pods**
* Use multi-container Pods **only when containers must share lifecycle**
* Do NOT put unrelated apps in one Pod
* One Pod = one main responsibility

# Hands-On

## 1. Single-Container Pod (The Standard)

It is a single unit of work with one clear responsibility.

### YAML Example (`single-pod.yaml`)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-single-pod
spec:
  containers:
  - name: nginx-container
    image: nginx:latest
    ports:
    - containerPort: 80

```

**Why use this?**

* **Isolation:** If the container crashes, only this Pod is affected.
* **Scaling:** You can scale these Pods independently without wasting resources on unnecessary helper containers.



## 2. Multi-Container Pod (The Helper Pattern)

This is used when you have a "Main" app that needs a "Helper" (like a log exporter or a local proxy) to function. They are tightly coupled and live/die together.

### YAML Example (`multi-pod.yaml`)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: sidecar-pod
spec:
  # Both containers share this volume
  volumes:
  - name: shared-logs
    emptyDir: {}

  containers:
  # 1. Main Application Container
  - name: main-app
    image: alpine
    command: ["/bin/sh", "-c"]
    args: ["while true; do date >> /var/log/app.log; sleep 5; done"]
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log

  # 2. Sidecar (Helper) Container
  - name: log-sidecar
    image: busybox
    command: ["/bin/sh", "-c"]
    args: ["tail -f /var/log/app.log"]
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log

```

**Key Differences in the YAML:**

* **`containers:` List:** In the multi-container version, the `containers` section is a list (denoted by multiple `- name:` entries).
* **`volumes:`:** Multi-container pods almost always use a shared volume (like `emptyDir`) to exchange data between the containers.


## 3. Hands-on Comparison Commands

To see the difference in your cluster, apply both and compare the **READY** column:

```bash
# Create them
kubectl apply -f single-pod.yaml
kubectl apply -f multi-pod.yaml

# Compare status
kubectl get pods

```

| Pod Name | Ready | Status |
| --- | --- | --- |
| `nginx-single-pod` | **1/1** | Running |
| `sidecar-pod` | **2/2** | Running |

> **Note:** The `2/2` in the Ready column means both containers inside that single Pod are healthy and running.
