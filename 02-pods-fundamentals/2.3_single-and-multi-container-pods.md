# Single-Container and Multi-Container Pods

## 1. What Is a Single-Container Pod?

A **single-container Pod** runs **one container** inside it.

This is:

* The **most common**
* The **recommended default**
* Easier to manage and debug

### When to Use

* Web applications
* APIs
* Microservices
* Background workers

### Simple Example

One Pod → One container → One responsibility


## 2. Create a Single-Container Pod (Hands-on)

### Command

```bash
kubectl run nginx-pod --image=nginx
```

### Verify

```bash
kubectl get pods
kubectl describe pod nginx-pod
```



## 3. What Is a Multi-Container Pod?

A **multi-container Pod** runs **more than one container** inside the same Pod.

All containers:

* Share the same IP
* Share the same volumes
* Start and stop together

### Important Rule

> Containers in a Pod are **tightly coupled**



## 4. Why Multi-Container Pods Exist

Multi-container Pods are used when containers must:

* Work together closely
* Share data or network
* Have the same lifecycle

They are **not** used for unrelated services.



## 5. Common Multi-Container Pod Patterns

### 1. Sidecar Pattern

* Main app container
* Helper container (logging, monitoring)

Example:

* App + log collector



### 2. Ambassador Pattern

* App container
* Proxy container

Example:

* App + API gateway

### 3. Adapter Pattern

* App container
* Data transformer container

Example:

* App + metrics formatter



## 6. Simple Multi-Container Pod Concept (No YAML Yet)

```
Pod
 ├── App Container
 └── Helper Container
```

Both containers:

* Use `localhost`
* Share storage
* Restart together


## 7. Verify Containers Inside a Pod

```bash
kubectl get pods
kubectl describe pod <pod-name>
```

To see container logs:

```bash
kubectl logs <pod-name> -c <container-name>
```


## 8. Single vs Multi-Container Pods (Quick View)

| Aspect      | Single-Container | Multi-Container  |
| ----------- | ---------------- | ---------------- |
| Complexity  | Low              | Medium           |
| Use case    | Most workloads   | Special cases    |
| Debugging   | Easy             | Slightly complex |
| Recommended | Yes              | Only when needed |


## 9. Best Practices (Very Important)

* Prefer **single-container Pods**
* Use multi-container Pods **only when containers must share lifecycle**
* Do NOT put unrelated apps in one Pod
* One Pod = one main responsibility



## 10. Key Takeaways

* A Pod can run **one or multiple containers**
* Single-container Pods are the default choice
* Multi-container Pods support helper patterns
* Containers in a Pod are **not independent**
