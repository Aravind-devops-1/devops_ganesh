# Pod Debugging Basics

## 1. Why Pod Debugging Is Important

Pods fail **very often** in real Kubernetes environments due to:

* Wrong images
* Application crashes
* Configuration issues
* Resource limits
* Networking problems

A DevOps engineer must quickly answer:

> ❝ Why is this Pod not running? ❞



## 2. First Rule of Pod Debugging

> **Never guess — always observe**

Kubernetes gives you **events, status, and logs**.
Debugging always follows the same order.



## 3. Basic Pod Status Check

### Check Pod Status

```bash
kubectl get pods
```

Common states you’ll see:

* `Pending`
* `Running`
* `CrashLoopBackOff`
* `ImagePullBackOff`
* `Error`

This tells **what is wrong**, not **why**.



## 4. Describe Pod (MOST IMPORTANT COMMAND)

```bash
kubectl describe pod <pod-name>
```

This command shows:

* Events
* Scheduling errors
* Image pull errors
* Probe failures
* Resource issues

 **Always check the `Events` section at the bottom**



## 5. Checking Container Logs

### View Logs

```bash
kubectl logs <pod-name>
```

If pod has multiple containers:

```bash
kubectl logs <pod-name> -c <container-name>
```

Use logs when:

* Pod is crashing
* App is exiting
* App starts but fails



## 6. Executing Into a Running Pod

If the Pod is **Running** but misbehaving:

```bash
kubectl exec -it <pod-name> -- /bin/sh
```

Or (for bash):

```bash
kubectl exec -it <pod-name> -- /bin/bash
```

Use this to:

* Check files
* Test ports
* Run curl
* Inspect environment variables


## 7. Common Pod Errors & Fix Direction

### CrashLoopBackOff

Meaning:

* App starts → crashes → restarts repeatedly

Check:

```bash
kubectl logs <pod-name>
```



### ImagePullBackOff

Meaning:

* Image not found or access denied

Check:

* Image name
* Tag
* Registry access



### Pending

Meaning:

* Pod not scheduled

Check:

```bash
kubectl describe pod <pod-name>
```

Look for:

* Insufficient CPU / memory
* Node issues



## 8. Events – The Hidden Gold

View cluster events:

```bash
kubectl get events --sort-by=.metadata.creationTimestamp
```

Events often tell the **exact reason** of failure.

## 10. Debugging from Node Perspective (Intro)

To see where the pod is running:

```bash
kubectl get pods -o wide
```

Shows:

* Node name
* Pod IP

Useful for networking issues.



## 11. Common Beginner Mistakes

* Ignoring `kubectl describe`
* Checking logs before events
* Assuming Kubernetes is broken
* Deleting pod without understanding error

---

## 12. DevOps Debugging Mindset

* Kubernetes rarely “breaks”
* Misconfiguration causes most issues
* Events + logs = truth
* Debug fast, not blindly

---

## 13. Key Takeaways

*  Always start with pod status
*  `kubectl describe` is your best friend
*  Logs explain crashes
*  Events explain scheduling & image issues
*  Follow a fixed debugging flow
