# Pod Networking Basics

## 1. What Is Pod Networking?

Pod networking is the underlying fabric that allows components in a cluster to talk. Kubernetes follows a "IP-per-Pod" model.

### The Three Layers of Communication:

1. **Intra-Pod:** Container-to-container (inside the same Pod).
2. **Inter-Pod:** Pod-to-Pod communication (same node or different nodes).
3. **Pod-to-Service:** Pod communication with the outside world or internal abstractions.


## 2. Pod IP Address (The "Single IP" Rule)

In Kubernetes, an IP address is assigned to the **Pod**, not the individual containers.

* **Shared Namespace:** All containers in a Pod share the same Network Namespace (the same IP and port space).
* **Port Conflicts:** Because they share the same IP, two containers in the same Pod **cannot** bind to the same port (e.g., you cannot have two containers both using port `80`).



## 3. Communication Patterns

### A. Intra-Pod (Container-to-Container)

Containers communicate via `localhost`. Think of them as different processes running on the same physical machine.

> **Example:**
> A "Helper" container can scrape metrics from an "App" container using:
> `curl http://localhost:8080`

### B. Inter-Pod (Pod-to-Pod)

The **Kubernetes Network Model** guarantees that:

1. All Pods can communicate with all other Pods without NAT.
2. Agents (like `kubelet`) can communicate with all Pods on that node.

```mermaid

    PodA[Pod A: 10.244.0.5] -- Direct IP --> PodB[Pod B: 10.244.1.7]
```

## 4. Why Pod IPs Are "Ephemeral"

**Crucial Concept:** Never hardcode a Pod IP.

* **Volatile:** If a Pod crashes and restarts, it may get a new IP.
* **Dynamic:** When scaling a Deployment, new Pods get random IPs from the CNI range.
* **Solution:** Use **Services** (a stable DNS name/IP that points to your Pods).


## 5. Implementation: The CNI

Kubernetes doesn't provide networking itself; it uses the **Container Network Interface (CNI)**.

* **Flannel:** Simple, overlay network.
* **Calico:** Advanced, supports Network Policies (firewalls).
* **Cilium:** High performance using eBPF.



## Hands-On: Networking Lab

### 1. Identify Pod IP and Infrastructure

Create an Nginx Pod and look at the "Wide" output:

```bash
kubectl run nginx-pod --image=nginx
kubectl get pod nginx-pod -o wide
```

*Take note of the `IP` and the `NODE` columns. This shows you exactly where the Pod lives in the cluster.*

### 2. Verify Shared Namespace

Exec into the Pod and check the network interfaces:

```bash
kubectl exec -it nginx-pod -- ip addr
```

You will see a `loopback` interface and an `eth0` interface with the Pod IP.

### 3. Test Connectivity (The "Ping" Test)

Create a temporary "Tester" Pod:

```bash
kubectl run busybox --image=busybox -i --tty --rm -- sh
```

*Note: The `--rm` flag deletes the pod automatically once you exit.*

Inside the Busybox shell:

```bash
# Replace with the IP from step 1
ping <NGINX_POD_IP> 
wget -qO- <NGINX_POD_IP>

```


## Common Pitfalls

* **Port Conflicts:** Trying to run two containers on port `80` in one Pod.
* **Firewalls:** Forgetting that `NetworkPolicies` might be blocking Pod-to-Pod traffic in production clusters.
* **Hardcoding:** Using `10.x.x.x` in a configuration file instead of a Service name.


### Summary Checklist

* Every Pod gets a unique IP.
* Containers in one Pod share an IP and use `localhost`.
* Pods can talk to each other across nodes without NAT.
* Pod IPs are temporary; Services are permanent.
