# Pod Best Practices and Anti-Patterns

## 1. Introduction

Pods are the **smallest deployable unit** in Kubernetes.
Following best practices ensures **reliability, scalability, and maintainability**.
Avoiding anti-patterns prevents **common operational issues**.



## 2. Best Practices

### 2.1 Keep Pods Small and Focused

* One main container per Pod
* Add sidecars only if necessary (logging, monitoring)
* Helps with:

  * Easier scaling
  * Simpler debugging

### 2.2 Use Labels Wisely

* Add meaningful `app`, `env`, or `tier` labels
* Enables:

  * Service selection
  * Monitoring & alerting
  * Autoscaling

### 2.3 Set Resource Requests and Limits

* Always define `cpu` and `memory` limits
* Prevents:

  * Node resource starvation
  * Pod eviction due to overuse

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

### 2.4 Use Readiness and Liveness Probes

* Ensures Pods are **healthy and ready to serve traffic**
* Reduces downtime during updates

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
```

### 2.5 Follow Naming Conventions

* Use lowercase, hyphen-separated names
* Helps with automation and readability



## 3. Common Anti-Patterns

| Anti-Pattern                               | Why It’s Bad                     | Suggested Fix                          |
| ------------------------------------------ | -------------------------------- | -------------------------------------- |
| Multiple unrelated containers in one Pod   | Hard to debug, scale, or restart | Use separate Pods + sidecars if needed |
| Hardcoding configuration inside containers | Difficult to update              | Use ConfigMaps / Secrets               |
| Not setting resource limits                | May starve nodes                 | Always define requests & limits        |
| Ignoring probes                            | Pods may serve unhealthy traffic | Add liveness & readiness probes        |
| Using Pod directly in production           | No auto-healing or scaling       | Use Deployments instead                |


## 4. Commands for Hands-On Validation

### View Pod Labels

```bash
kubectl get pods --show-labels
```

### Describe Pod for Resource and Probe Info

```bash
kubectl describe pod <pod-name>
```

### Check Pod Status

```bash
kubectl get pods -o wide
```


## 5. Key Takeaways

✔ Keep Pods **small, single-purpose, and observable**
✔ Always define **resources and probes**
✔ Use **labels** for organization & automation
✔ Avoid anti-patterns for **scalability & reliability**

# Kubernetes Pods Best Practices – Hands-On Lab

## 1. Objective

Learn and practice:

* Creating small, focused Pods
* Using labels and selectors
* Setting resource requests & limits
* Adding liveness and readiness probes
* Observing Pod behavior

## 2. Lab Environment

* Minikube running
* kubectl configured
* Docker images available (e.g., `nginx`, `busybox`)


## 3. Step 1 – Create a Pod with Labels

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: frontend-pod
  labels:
    app: frontend
    env: dev
spec:
  containers:
  - name: nginx
    image: nginx:latest
    ports:
    - containerPort: 80
```

**Commands:**

```bash
kubectl apply -f frontend-pod.yaml
kubectl get pods --show-labels
```

Observe the labels appear for your Pod.



## 4. Step 2 – Add Resource Requests and Limits

Update your Pod YAML:

```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

Apply the updated Pod:

```bash
kubectl apply -f frontend-pod.yaml
kubectl describe pod frontend-pod
```

Observe CPU and memory requests/limits under `Containers:` section.



## 5. Step 3 – Add Liveness and Readiness Probes

Update container spec:

```yaml
livenessProbe:
  httpGet:
    path: /
    port: 80
  initialDelaySeconds: 5
  periodSeconds: 5

readinessProbe:
  httpGet:
    path: /
    port: 80
  initialDelaySeconds: 5
  periodSeconds: 5
```

Apply changes:

```bash
kubectl apply -f frontend-pod.yaml
kubectl describe pod frontend-pod
```

Observe `Liveness` and `Readiness` probes in the description.



## 6. Step 4 – Test Pod Labels with a Selector

```bash
kubectl get pods -l app=frontend
kubectl get pods -l env=dev
kubectl get pods -l app=frontend,env=dev
```

Confirm that only your Pod is selected based on labels.


## 7. Step 5 – Simulate Anti-Patterns and Observe

### 7.1 Multiple unrelated containers

```yaml
containers:
- name: nginx
  image: nginx
- name: busybox
  image: busybox
```

* Apply and notice:

  * Harder to debug
  * Logs are mixed
  * Restart behavior may differ

### 7.2 Pod without probes

* Remove probes and observe readiness and liveness info
* kubectl may show the Pod as `Running` even if container fails


## 8. Step 6 – Cleanup

```bash
kubectl delete pod frontend-pod
```


## 9. Key Learning Points

* Always **keep Pods focused** (single main container + sidecar if needed)
* **Labels** are critical for selection and automation
* **Resource requests and limits** prevent cluster resource issues
* **Liveness and readiness probes** keep workloads healthy
* Anti-patterns demonstrate why these best practices matter
