# 06 – DaemonSet Fundamentals

A **DaemonSet** ensures that **a specific Pod runs on every Node (or selected Nodes)** in the cluster.


# 1. What Is a DaemonSet?

A **DaemonSet** is a Kubernetes workload object that guarantees:

* **One Pod per Node**
* **Automatic Pod placement on new Nodes**
* **Automatic Pod removal when Nodes are removed**

DaemonSets are mainly used for **node-level services**, not applications.

> **If a Node exists → the Pod must exist on that Node**

---

# 2. Why DaemonSets Exist

Some workloads are **node-specific**, not application replicas.

Deployments and StatefulSets:

* Scale by **number of Pods**
* Are **service-oriented**

DaemonSets:

* Scale by **number of Nodes**
* Are **infrastructure-oriented**

### Typical Problems Without DaemonSets

* Manually installing agents on each Node
* Forgetting to deploy agents on new Nodes
* Inconsistent node-level monitoring/logging

### What DaemonSets Solve

* Automatic Pod scheduling on **every Node**
* Zero manual Node management
* Consistent cluster-wide behavior

---

# 3. Core Concepts (How DaemonSet Works)

DaemonSet behavior is built on **four key principles**:

1. **One Pod Per Node**
2. **Node Awareness**
3. **Automatic Node Tracking**
4. **Controller-Managed Lifecycle**



   ## 1. One Pod Per Node

   DaemonSet ensures:

   ```
   1 Node = 1 Pod
   ```
   
   If the cluster has:
   
   | Nodes | Pods |
   | ----- | ---- |
   | 3     | 3    |
   | 5     | 5    |
   
   > You do **not** set replicas manually.
   
   ---
   
   ## 2. Node Awareness
   
   DaemonSet Pods are scheduled **directly onto Nodes**, bypassing normal replica logic.
   
   * Pod runs **on each Node**
   * Can be limited using:
   
     * `nodeSelector`
     * `nodeAffinity`
     * `tolerations`
   
   Example:
   
   * Run Pods **only on worker nodes**
   * Or **only on GPU nodes**
   
   ---
   
   ## 3. Automatic Node Tracking
   
   DaemonSet continuously watches the cluster:
   
   * New Node added → **Pod created automatically**
   * Node removed → **Pod deleted automatically**
   
   No manual intervention needed.
   
   ---
   
   ## 4. Controller-Managed Lifecycle
   
   DaemonSet Controller ensures:
   
   * Pod is recreated if deleted
   * Pod exists exactly once per eligible Node
   * Cluster stays in desired state

# 5. When to Use DaemonSet

Use DaemonSet when your workload needs:

* **One instance per Node**
* **Direct access to Node resources**
* **Cluster-wide consistency**

### Common Use Cases

* Log collectors
* Monitoring agents
* Security agents
* Network plugins
* Storage drivers

# 7. Minimal DaemonSet YAML

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-monitor
spec:
  selector:
    matchLabels:
      app: node-monitor
  template:
    metadata:
      labels:
        app: node-monitor
    spec:
      containers:
      - name: monitor
        image: busybox
        command: ["sh", "-c", "while true; do echo monitoring; sleep 30; done"]
```

> **No `replicas` field** — Kubernetes calculates it from Nodes.

---

# 8. Key Fields Explained

| Field                   | Purpose                      |
| ----------------------- | ---------------------------- |
| kind: DaemonSet         | Declares node-based workload |
| selector                | Links Pods to DaemonSet      |
| template                | Pod definition               |
| nodeSelector / affinity | Limit which Nodes get Pods   |
| tolerations             | Allow Pods on tainted Nodes  |

---

# 9. Scheduling Control (Important)

DaemonSets often run on **tainted Nodes**.

Example:

* Master / control-plane Nodes
* Infra Nodes

### Toleration Example

```yaml
tolerations:
- key: node-role.kubernetes.io/control-plane
  effect: NoSchedule
```

> Without tolerations, Pods will **not run on tainted Nodes**

---

# 10. Updating a DaemonSet

DaemonSets support **rolling updates**.

Default behavior:

* Update Pods **one Node at a time**
* Ensures cluster stability

```bash
kubectl rollout status daemonset node-monitor
```

---

# 11. Self-Healing Behavior

If a DaemonSet Pod crashes or is deleted:

* Controller recreates the Pod
* Pod is recreated **on the same Node**

```bash
kubectl delete pod <daemonset-pod>
```

Result:

```
Pod recreated automatically
```

---

# 12. Scaling Behavior

DaemonSets **do not scale manually**.

Scaling happens by:

* Adding Nodes → Pods increase
* Removing Nodes → Pods decrease

> **Scaling = Node count**

---

# 13. DaemonSet vs Deployment vs StatefulSet

| Feature            | Deployment | StatefulSet | DaemonSet |
| ------------------ | ---------- | ----------- | --------- |
| Pod identity       | No         | Yes         | No        |
| Stable storage     | No         | Yes         | No        |
| Ordered operations | No         | Yes         | No        |
| Scaling method     | Replicas   | Replicas    | Nodes     |
| One Pod per Node   | No         | No          | Yes       |
| Typical use        | Apps       | Databases   | Agents    |

---

# 14. Common Mistakes

* Trying to scale DaemonSet manually
* Using DaemonSet for applications
* Forgetting tolerations
* Running unnecessary Pods on all Nodes

---

# 15. Best Practices

1. Use DaemonSet only for **node-level services**
2. Limit Nodes using affinity or selectors
3. Add tolerations carefully
4. Monitor DaemonSet health
5. Avoid heavy workloads in DaemonSets

---

# 16. Key Takeaway

> **DaemonSet ensures cluster-wide consistency at the Node level**

If you understand DaemonSets, you understand:

* Node-based workloads
* Infrastructure agents
* Cluster-wide services
