# Kubernetes Scheduling Terms Explained

Kubernetes decides **where a Pod runs** using **Node labels, taints, tolerations, and affinity rules**.

---

# 1. Node Selector

### What is Node Selector?

**Node Selector** is the **simplest way** to tell Kubernetes:

> “Run this Pod only on Nodes with this label.”

It works by **matching Pod requirements with Node labels**.

### How it works

* Nodes have labels
* Pods request labels
* If labels match → Pod is scheduled

### Example

Node label:

```bash
# Label the Minikube Node
kubectl label node minikube disk=ssd

# verify
kubectl get nodes --show-labels
```

Pod spec:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod
spec:
  nodeSelector:
    disk: ssd
  containers:
  - name: nginx
    image: nginx:1.25
```

### Key Characteristics

* Scheduler checks node labels
* minikube node has disk=ssd
* Pod is scheduled successfully

### If Label Does NOT Exist

If you remove the label:

```bash
# Remove label
kubectl label node minikube disk-
```

Result:

```
Pod stays in Pending state
```

---

## 2. Node Affinity

### What is Node Affinity?

**Node Affinity** is an **advanced version of nodeSelector**.

It allows Pods to express:

* **Required rules** (must match)
* **Preferred rules** (best effort)

### Types of Node Affinity

| Type                                            | Meaning      |
| ----------------------------------------------- | ------------ |
| requiredDuringSchedulingIgnoredDuringExecution  | Must match   |
| preferredDuringSchedulingIgnoredDuringExecution | Try to match |

### Example

```bash
# Label the Minikube Node
kubectl label node node1 disk=ssd

# verify
kubectl get nodes --show-labels
```

### Complete Pod YAML with Node Affinity
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: affinity-demo-pod
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: disk
            operator: In
            values:
            - ssd
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
```

```bash
# Apply:
kubectl apply -f affinity-pod.yaml
```

### Key Characteristics

* Pod-driven
* Flexible rules
* Supports logical expressions
* Still cannot protect Nodes

# 3. Taints

### What is a Taint?

A **Taint** is a **Node-level rule** that says:

> “Do NOT schedule Pods here unless they explicitly allow it.”

Taints **repel Pods**.

## Generic Explanation

* **Taint** = “No entry” sign on a node
* **Scheduler** = Security guard

A **taint is applied to a node** to **repel Pods**.

By default, when a node has a taint:

> **No Pods are allowed to enter this node**

### Syntax

```
key=value:effect
```

Example:

```bash
kubectl taint nodes node1 role=infra:NoSchedule
```

### Effects

| Effect           | Behavior           |
| ---------------- | ------------------ |
| NoSchedule       | Block new Pods     |
| PreferNoSchedule | Avoid Pods         |
| NoExecute        | Evict running Pods |

### 1. Taint the Node

Minikube has a single node named `minikube`.

```bash
kubectl taint node minikube env=prod:NoSchedule
```
Verify:

```bash
kubectl describe node minikube | grep -i taint
```
Expected:

```
env=prod:NoSchedule
```
---

### 2. Pod WITHOUT Toleration (Will Fail)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: no-toleration-pod
spec:
  containers:
  - name: nginx
    image: nginx:1.25
```
Apply:

```bash
kubectl apply -f no-toleration.yaml
kubectl get pods
```
Result:
```
no-toleration-pod   Pending
```

Reason:

> Node is tainted and Pod has **no permission** to run there.

---

### 3. Pod WITH Toleration (Will Work)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: toleration-pod
spec:
  tolerations:
  - key: "env"
    operator: "Equal"
    value: "prod"
    effect: "NoSchedule"
  containers:
  - name: nginx
    image: nginx:1.25
```

Apply:

```bash
kubectl apply -f toleration.yaml
kubectl get pods
```
Result:

```
toleration-pod   Running
```
---

### 4. Remove the Taint (Cleanup)

```bash
kubectl taint node minikube env=prod:NoSchedule-
```

Verify:

```bash
kubectl describe node minikube | grep -i taint
```

# 4. Tolerations

### What is a Toleration?

A **Toleration** is a **Pod-level permission** that says:

> “This Pod is allowed to run on tainted Nodes.”

Tolerations **do not schedule Pods**, they only **allow them**.

### Generic Explanation
* **Taint** = “No entry” sign on a node
* **Toleration** = “Special permission badge”
* **Scheduler** = Security guard

Having a badge (**toleration**) does NOT mean:

> You will be sent to that building

It only means:

> You are **allowed in if you are sent there**

---
### Add a Taint to the Minikube Node

Minikube runs as a **single node** named `minikube`.

Add a taint:

```bash
kubectl taint nodes minikube env=prod:NoSchedule
```

Meaning:

* `env=prod` → taint key/value
* `NoSchedule` → Pods **will not be scheduled** unless they tolerate it

---

### Verify the Taint

```bash
kubectl describe node minikube | grep -i taint
```

Expected output:

```
Taints: env=prod:NoSchedule
```

---

### Pod WITHOUT Toleration (Will Fail)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: no-toleration-pod
spec:
  containers:
  - name: nginx
    image: nginx
```

Apply it:

```bash
kubectl apply -f no-toleration.yaml
```

Check status:

```bash
kubectl get pod no-toleration-pod
```

Result:

```
Pending
```

Reason:

> Pod does NOT tolerate the taint → Scheduler blocks it

---

### Pod WITH Toleration (Will Work)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: toleration-pod
spec:
  tolerations:
  - key: "env"
    operator: "Equal"
    value: "prod"
    effect: "NoSchedule"
  containers:
  - name: nginx
    image: nginx
```

Apply it:

```bash
kubectl apply -f toleration-pod.yaml
```

Check status:

```bash
kubectl get pod toleration-pod
```

Result:

```
Running
```

---

### What Actually Happened (Beginner View)

* Node has **“No entry” sign** (taint)
* First Pod →  no permission → blocked
* Second Pod →  has permission (toleration) → allowed

But remember:

> **Toleration does NOT force scheduling**
> It only **allows** the Pod to be scheduled **if the scheduler chooses that node**

---

### Clean Up (Important in Minikube)

Remove the taint to avoid blocking future Pods:

```bash
kubectl taint nodes minikube env=prod:NoSchedule-
```

---

## How These Work Together (Very Important)

### Scheduling Flow (Simplified)

1. Node Selector / Node Affinity filters Nodes
2. Taints reject Pods
3. Tolerations allow Pods onto tainted Nodes
4. Scheduler selects best Node

---

## Comparison Table (Very Clear)

| Feature            | Node Selector | Node Affinity | Taints | Tolerations  |
| ------------------ | ------------- | ------------- | ------ | ------------ |
| Applied on         | Pod           | Pod           | Node   | Pod          |
| Controls placement | Yes           | Yes           | Yes    | No           |
| Rejects Pods       | No            | No            | Yes    | No           |
| Allows Pods        | No            | No            | No     | Yes          |
| Logic support      | Exact match   | Advanced      | Strict | Match taints |
| Protects Node      | No            | No            | Yes    | No           |

---

## When to Use What

| Scenario            | Best Choice   |
| ------------------- | ------------- |
| Simple placement    | Node Selector |
| Complex placement   | Node Affinity |
| Protect Nodes       | Taints        |
| Allow specific Pods | Tolerations   |

---

## Easy Mental Model

* **Node Selector** → “Place Pod here”
* **Node Affinity** → “Prefer / require Pod here”
* **Taint** → “Keep Pods out”
* **Toleration** → “Exception pass”

# Summary

Here is a **short, simple, beginner-friendly version** with the same meaning and clean flow.

---

## Pod Scheduling Flow (Simple)

### Phase 1: Scheduler Filters (Pull)

The scheduler checks **what the Pod asks for** and finds matching nodes.

* **NodeSelector**
  Pod runs **only on nodes with matching labels**.
  No match → Pod stays **Pending**.

* **Node Affinity (Hard)**
  Mandatory rules (`requiredDuringSchedulingIgnoredDuringExecution`).
  Rule fails → Pod is **rejected**.

* **Node Affinity (Soft)**
  Preferred rules (`preferredDuringSchedulingIgnoredDuringExecution`).
  Match gives **higher priority**, but Pod can still run if not matched.

---

### Phase 2: Node Enforcement (Push)

The **node decides** whether to accept the Pod.

* **No taints on node** → Pod is scheduled.
* **Node has taints** → Pod must have a **matching toleration**.
* **With toleration** → Pod allowed.
* **Without toleration** → Pod rejected.

---

### One-Line Summary

> **Scheduler finds suitable nodes, but taints decide final entry.**


### Summary Table: How to Read the Flow

| Logic Step | Source | Type | Effect if Fail |
| --- | --- | --- | --- |
| **Node Selector** | Pod | Filter | Stays Pending |
| **Hard Affinity** | Pod | Filter | Stays Pending |
| **Soft Affinity** | Pod | Preference | Schedules anyway (lower priority) |
| **Taints** | Node | Restriction | Rejected by the Node |
