# Kubernetes Architecture – Core Concepts Explained

## 1. Introduction

Kubernetes (K8s) is a **container orchestration platform** designed to manage:

* Deployment
* Scaling
* Networking
* Self-healing
  of containerized applications.

To understand Kubernetes deeply (and use it confidently in DevOps), you must first understand **its architecture**.


## 2. High-Level Kubernetes Architecture

Kubernetes follows a **Master–Worker (Control Plane–Node)** architecture.

```
+---------------------------+
|  Control Plane            |
|---------------------------|
| API Server                |
| Scheduler                 |
| Controller Manager        |
| etcd                      |
| Cloud Controller Manager  |
+----------+----------------+
           |
           |
+----------+----------------+
| Worker Nodes              |
|---------------------------|
| kubelet                   |
| kube-proxy                |
| Container Runtime         |
| Pods                      |
+---------------------------+
```

## 3. Control Plane Components (Brain of Kubernetes)

The **Control Plane** makes global decisions about the cluster.

### 3.1 API Server (kube-apiserver)

**Role:**

* Entry point to the Kubernetes cluster
* All requests go through API Server (kubectl, UI, CI/CD)

**Key Points:**

* Exposes REST APIs
* Authenticates and validates requests
* Talks to etcd

Example:

```bash
kubectl get pods
```

➡ Request → API Server → Response



### 3.2 etcd (Cluster Database)

**Role:**

* Key-value store
* Stores entire cluster state

**Stores:**

* Pods
* Deployments
* ConfigMaps
* Secrets
* Node information

**Important:**

* If etcd is lost → cluster state is lost
* Backup is critical in production



### 3.3 Scheduler (kube-scheduler)

**Role:**

* Decides **which pod runs on which node**

**Decision based on:**

* CPU & Memory availability
* Node selectors
* Taints and tolerations
* Affinity rules

**Does NOT run pods**
➡ Only assigns pods to nodes


### 3.4 Controller Manager (kube-controller-manager)

**Role:**

* Ensures **desired state = actual state**

**Examples of Controllers:**

* Node Controller
* ReplicaSet Controller
* Deployment Controller

Example:

* Desired replicas = 3
* One pod crashes
* Controller creates a new pod automatically



## 4. Worker Node Components (Execution Layer)

Worker nodes are where **applications actually run**.


### 4.1 kubelet

**Role:**

* Agent running on each node
* Communicates with API Server
* Ensures containers are running as expected

**Responsibilities:**

* Starts pods
* Restarts failed containers
* Reports node health


### 4.2 Container Runtime

**Role:**

* Runs containers inside pods

**Common runtimes:**

* Docker (via containerd)
* containerd
* CRI-O

Minikube default:

```
containerd
```


### 4.3 kube-proxy

**Role:**

* Handles networking and service routing

**Functions:**

* Implements Services (ClusterIP, NodePort)
* Load balances traffic to pods
* Manages iptables / IPVS rules

## 5. Pod – Smallest Deployable Unit

### What is a Pod?

A **Pod** is:

* One or more containers
* Shared network (same IP)
* Shared storage (volumes)

Example:

* App container
* Sidecar container (logging, proxy)

## 6. How Components Work Together (Flow)

### Pod Creation Flow

1. Developer runs:

   ```bash
   kubectl apply -f pod.yaml
   ```
2. API Server receives request
3. Scheduler assigns node
4. Controller Manager monitors state
5. kubelet pulls image & starts container
6. kube-proxy routes traffic

## 7. Kubernetes Objects (High-Level)

| Object     | Purpose            |
| ---------- | ------------------ |
| Pod        | Run containers     |
| Deployment | Manage replicas    |
| ReplicaSet | Maintain pod count |
| Service    | Expose application |
| ConfigMap  | Configuration      |
| Secret     | Sensitive data     |

## 8. Single-Node vs Multi-Node Cluster

### Minikube (Learning)

* Control Plane + Worker on same node
* Used for development & learning

### Production Cluster

* Multiple control plane nodes
* Multiple worker nodes
* High availability (HA)

## 9. DevOps Perspective (Why This Matters)

Understanding architecture helps you:

* Debug pod failures
* Fix scheduling issues
* Optimize resource usage
* Design HA systems
* Crack Kubernetes interviews

## 10. Quick Summary

* API Server is the gateway
* etcd stores cluster state
* Scheduler decides pod placement
* Controller Manager maintains desired state
* kubelet runs workloads
* kube-proxy handles networking
