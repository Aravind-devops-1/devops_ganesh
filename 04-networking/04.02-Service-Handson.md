# Hands-on: Kubernetes Service Object


### 1. Prerequisite: Create a Simple Pod

Before creating a Service, we need a **Pod** that the Service can send traffic to.

### Pod YAML: `web-pod.yaml`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-pod
  labels:
    app: web
spec:
  containers:
  - name: nginx
    image: nginx
    ports:
    - containerPort: 80
```

---

### Easy Explanation of Pod Fields

* **kind: Pod** :  Creates a single Pod
* **metadata.name** :  Name of the Pod
* **metadata.labels** :  Labels identify the Pod
   Services use these labels to connect to Pods
* **containers.name** :  Name of the container inside the Pod
* **image: nginx** :  Runs an NGINX web server
* **containerPort: 80** : Port on which the application listens inside the Pod

---

### Create the Pod

```bash
kubectl apply -f web-pod.yaml
```

### Verify Pod

```bash
kubectl get pods
kubectl describe pod web-pod
```

---

### Why This Pod Is Important

* The label `app: web`
* Matches the Service selector:

  ```yaml
  selector:
    app: web
  ```
* This is how **Service → Pod mapping works**
---

## Service Configuration Explained in the below order
1. Service without targetPort
1. Service with targetPort
1. Service with Named Port
1. Service with Multiple Ports
1. Service Without Selector

## 1. Basic Service Configuration 

 file-name : `service-basic.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
  ports:
  - port: 80
```

### What each field means

* **apiVersion: v1**
  Uses core Kubernetes API
* **kind: Service**
  Tells Kubernetes to create a Service object
* **metadata.name**
  Name of the Service (used inside the cluster)
* **spec.selector**
  Tells the Service **which Pods to send traffic to**
  → Matches Pods with label `app=web`
* **spec.ports.port**
  Port on which the Service is available

---

### Commands

```bash
kubectl apply -f service-basic.yaml
kubectl get services
kubectl describe service web-service
```

---

## 2. Service with targetPort

file-name : `service-target-port.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
```

### Easy Explanation

* **port**
  Port exposed by the Service
* **targetPort**
  Port on the Pod/container where the application is actually running

Example:

* Service listens on **80**
* Pod app listens on **8080**

---

### Commands

```bash
kubectl apply -f service-target-port.yaml
kubectl describe service web-service
```

---

## 3. Service with Named Port

file-name : `service-named-port.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
spec:
  selector:
    app: web
  ports:
  - name: http
    port: 80
    targetPort: http
```

Pod container should have:

```yaml
ports:
- name: http
  containerPort: 8080
```

### Why this is useful

* Makes configuration **more readable**
* Avoids confusion when ports change
* Common in production setups

---

## 4. Service with Multiple Ports

file-name : `service-multi-port.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: app-service
spec:
  selector:
    app: web
  ports:
  - name: http
    port: 80
    targetPort: 8080
  - name: metrics
    port: 9090
    targetPort: 9090
```

### Real-World Use Case

* One port for application traffic
* Another port for monitoring / metrics

---

## 5. Service Without Selector

file-name : `service-no-selector.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-service
spec:
  ports:
  - port: 80
```

### Simple Explanation

* Service **does not automatically connect to Pods**
* Used when endpoints are managed manually
* Common in special or external integrations

---

## 6. Common kubectl Commands for Services

```bash
kubectl get svc
kubectl describe svc <service-name>
kubectl delete svc <service-name>
```

---

## Summary of Service fields

* A **Service connects users to Pods**
* Services use **labels and selectors**
* `port` = Service port
* `targetPort` = Pod/container port
* YAML structure stays mostly the same across all Services

# Hands-on: Kubernetes Service – **ClusterIP**


## Below Pod Used for All Examples



```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-pod
  labels:
    app: web
spec:
  containers:
  - name: nginx
    image: nginx
    ports:
    - containerPort: 80
```

---

## 1. Basic ClusterIP Service

 file-name : `svc-clusterip-basic.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-clusterip
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - port: 80
```

### Easy Explanation

* **type: ClusterIP**
  Exposes the Service **inside the cluster only**
* **selector.app: web**
  Matches Pods with label `app=web`
* **port: 80**
  Port exposed by the Service

Kubernetes automatically assigns:

* Cluster IP (virtual IP)
* Endpoint mapping to Pods

---

### Commands

```bash
kubectl apply -f svc-clusterip-basic.yaml
kubectl get svc
kubectl describe svc web-clusterip
```

---

## 2. ClusterIP with targetPort

 file-name : `svc-clusterip-targetport.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-clusterip
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
```

### Easy Explanation

* **port** :  Port used by clients inside the cluster
* **targetPort** :  Actual port where the container is listening

If `targetPort` is omitted, Kubernetes assumes it equals `port`

---

## 3. ClusterIP with Named Ports

### Step 1: Pod with Named Port

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-pod
  labels:
    app: web
spec:
  containers:
  - name: nginx
    image: nginx
    ports:
    - name: http
      containerPort: 80
```
```

---

 file-name : `svc-clusterip-named.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-clusterip
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - name: http
    port: 80
    targetPort: http
```

### Why This Matters

* Improves readability
* Avoids port mismatch issues
* Widely used in production

---

## 4. ClusterIP with Multiple Ports

 file-name : `svc-clusterip-multiport.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: app-clusterip
spec:
  type: ClusterIP
  selector:
    app: web
  ports:
  - name: http
    port: 80
    targetPort: 80
  - name: metrics
    port: 9113
    targetPort: 9113
```
 file-name : multi-port-pod.yaml

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-pod
  labels:
    app: web
spec:
  containers:
  - name: web-app
    image: nginx
    ports:
    - name: http
      containerPort: 80
  - name: metrics
    image: prom/statsd-exporter
    ports:
    - name: metrics
      containerPort: 9113
```
### Real-World Use Case

* App traffic on port 80
* Metrics scraped by Prometheus on another port

---

## 5. Headless ClusterIP (ClusterIP: None)

 file-name : `svc-clusterip-headless.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-headless
spec:
  clusterIP: None
  selector:
    app: web
  ports:
  - port: 80
```

### Easy Explanation

* **No virtual IP**
* DNS returns **Pod IPs directly**
* Used for:

  * Stateful apps
  * Direct Pod communication

---

## 6. ClusterIP Without Selector

 file-name : `svc-clusterip-noselector.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: manual-service
spec:
  type: ClusterIP
  ports:
  - port: 80
```

### What This Means

* Service does **not automatically connect to Pods**
* Endpoints must be defined manually
* Used when:
  * Connecting to external services
  * Advanced networking setups

---

## 7. Common Commands for ClusterIP

```bash
kubectl get svc
kubectl describe svc web-clusterip
kubectl get endpoints web-clusterip
kubectl delete svc web-clusterip
```

---

## Summary of ClusterIP 

* **ClusterIP is the default Service**
* Used for **internal communication only**
* Uses **labels + selectors**
* Supports:
  * targetPort
  * named ports
  * multiple ports
  * headless mode

# NodePort Service – Pod Configuration

## 1. NodePort Service Configuration

file-name : `app-nodeport.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: app-nodeport
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - name: http
    port: 80
    targetPort: 80
    nodePort: 30080
  - name: metrics
    port: 9113
    targetPort: 9113
    nodePort: 30911
```
Matching Pod Configuration
* file-name : `web-pod.yaml`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-pod
  labels:
    app: web
spec:
  containers:
  - name: web-app
    image: nginx
    ports:
    - name: http
      containerPort: 80

  - name: metrics
    image: prom/statsd-exporter
    ports:
    - name: metrics
      containerPort: 9113
```
---

### Easy Explanation (Service Side)

* **type: NodePort**
  Exposes the Service on a port of every node
* **selector.app: web**
  Connects Service to Pods with label `app=web`
* **port**
  Internal Service port
* **targetPort**
  Pod/container port
* **nodePort**
  External port on the Node
  (Must be between `30000–32767`)

---

## 2. Why This Pod Works with the NodePort Service

### 1️. Label Matching

```yaml
labels:
  app: web
```

* Matches Service selector
* Service automatically discovers the Pod

---

### 2️. Port Mapping

| Service Port Name | nodePort | targetPort | Pod containerPort |
| ----------------- | -------- | ---------- | ----------------- |
| http              | 30080    | 80         | 80                |
| metrics           | 30911    | 9113       | 9113              |

Traffic flow:

```
NodeIP:30080  → Service → web-app container
NodeIP:30911  → Service → metrics container
```

---

## 3. Create and Verify

```bash
kubectl apply -f web-pod.yaml
kubectl apply -f app-nodeport.yaml
```

Verify:

```bash
kubectl get pods
kubectl get svc app-nodeport
kubectl get endpoints app-nodeport
```

---

## 4. Access the Application

### Get Node IP

```bash
kubectl get nodes -o wide
```

### Access Services

```text
http://<NodeIP>:30080     → Web App
http://<NodeIP>:30911     → Metrics
```

(For Minikube, use `minikube ip`)

# Hands-on: Kubernetes Service – LoadBalancer

## What You Will Learn

* How a LoadBalancer Service works
* How to expose applications externally
* Different valid Service configurations for LoadBalancer
* How Services connect to Pods using selectors and ports

---

## Prerequisite: Pod Used for All Examples

This Pod will be reused for all Service configurations.

### Pod YAML: `web-pod.yaml`

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-pod
  labels:
    app: web
spec:
  containers:
  - name: web
    image: nginx
    ports:
    - name: http
      containerPort: 80
    - name: metrics
      containerPort: 9113
```

Create the Pod:

```bash
kubectl apply -f web-pod.yaml
kubectl get pods
```

---

## 1. Basic LoadBalancer Service Configuration

### Service YAML: `svc-lb-basic.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: app-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80
```

### Explanation

* `type: LoadBalancer`
  Requests an external load balancer from the cloud provider
* `selector`
  Matches Pods with label `app=web`
* `port`
  Port exposed by the Service

Kubernetes automatically:

* Assigns a ClusterIP
* Creates NodePorts internally
* Provisions an external IP (cloud environments)

---

### Verify

```bash
kubectl get svc app-lb
```

Look for `EXTERNAL-IP`.

---

## 2. LoadBalancer Service with targetPort

### Service YAML: `svc-lb-targetport.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: app-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
```

### Explanation

* `port`
  External Service port
* `targetPort`
  Container port inside the Pod

If `targetPort` is omitted, Kubernetes assumes it is the same as `port`.

---

## 3. LoadBalancer Service with Named Port

### Service YAML: `svc-lb-named.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: app-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - name: http
    port: 80
    targetPort: http
```

### Explanation

* Named ports improve readability
* Prevent port mismatch errors
* Recommended best practice

The Service routes traffic to the container port named `http`.

---

## 4. LoadBalancer Service with Multiple Ports

### Service YAML: `svc-lb-multiport.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: app-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - name: http
    port: 80
    targetPort: http
  - name: metrics
    port: 9113
    targetPort: metrics
```

### Explanation

* One LoadBalancer
* Multiple exposed ports
* Common for applications with metrics endpoints

Traffic routing:

* Port 80 → web container
* Port 9113 → metrics container

---

## 5. LoadBalancer Service Without Selector

This pattern exists but is **rare and advanced**.

### Service YAML: `svc-lb-noselector.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-lb
spec:
  type: LoadBalancer
  ports:
  - port: 80
```

### Explanation

* No selector means:
  * Service does not automatically discover Pods
* Endpoints must be created manually
* Used when routing to:
  * External services
  * Legacy systems

Pods are not auto-linked in this case.

---

## How LoadBalancer Traffic Flows

```
Client
  ↓
External Load Balancer IP
  ↓
NodePort (auto-created)
  ↓
ClusterIP Service
  ↓
Pod
```

---

## Common Commands

```bash
kubectl apply -f svc-lb-basic.yaml
kubectl get svc
kubectl describe svc app-lb
kubectl get endpoints app-lb
```

To delete:

```bash
kubectl delete svc app-lb
```

---

## Important Beginner Notes

* LoadBalancer works best in cloud environments (AWS, GCP, Azure)
* In Minikube or Kind, external IP may stay pending
* Internally, LoadBalancer always uses ClusterIP and NodePort
* LoadBalancer is often combined with Ingress in production

---

## Beginner Takeaway

* LoadBalancer exposes applications publicly
* Uses labels and selectors like other Services
* Supports:
  * targetPort
  * named ports
  * multiple ports
* Selector-less usage is advanced and uncommon


# Hands-on: Kubernetes Service – ExternalName (DNS Alias)

## What You Will Learn

* What an ExternalName Service is
* How Kubernetes uses DNS aliasing
* When to use ExternalName instead of selectors
* Valid Service configurations for ExternalName

---

## Key Concept (Very Important)

**ExternalName Service does NOT:**

* Create ClusterIP
* Create Endpoints
* Use selectors
* Proxy traffic

It only creates a **DNS alias**.

---

## When to Use ExternalName

* Access external services using Kubernetes DNS
* Abstract service endpoints
* Migrate services without changing application code
* Point to managed services (DB, API, SaaS)

---

## 1. Basic ExternalName Service Configuration

### Service YAML: `svc-externalname-basic.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-api
spec:
  type: ExternalName
  externalName: api.example.com
```

### Explanation

* `type: ExternalName`
  Declares a DNS alias service
* `externalName`
  Fully qualified domain name of the external service

This creates a DNS record:

```text
external-api.default.svc.cluster.local → api.example.com
```

---

## How DNS Resolution Works

When a Pod queries:

```text
external-api.default.svc.cluster.local
```

DNS returns:

```text
api.example.com
```

The application connects **directly** to the external service.

---

## 2. ExternalName with Port Declaration (Documentation Only)

Kubernetes ignores ports for routing, but they can be added for clarity.

### Service YAML: `svc-externalname-withport.yaml`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  type: ExternalName
  externalName: db.example.com
  ports:
  - port: 5432
```

### Explanation

* `ports` here are **informational**
* DNS behavior is unchanged
* Some tools expect ports for consistency

---

## 3. ExternalName for Internal-to-External Migration

### Use Case

You want to migrate from an internal service to a managed external service **without changing app code**.

### Before (ClusterIP Service)

```yaml
name: payment-service
```

### After (ExternalName Service)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: payment-service
spec:
  type: ExternalName
  externalName: payments.prod.example.com
```

Applications continue using:

```text
payment-service.default.svc.cluster.local
```

---

## What Is NOT Supported (Excluded by Design)

The following are **not valid** for ExternalName and are intentionally excluded:

* Selector
* targetPort
* Multiple ports (functional)
* Endpoints
* Load balancing

---

## Common Commands

```bash
kubectl apply -f svc-externalname-basic.yaml
kubectl get svc
kubectl describe svc external-api
```

Check DNS from a Pod:

```bash
kubectl run dns-test --image=busybox -it --rm -- sh
nslookup external-api
```

---

## How ExternalName Traffic Flows

```
Pod
  ↓
Kubernetes DNS
  ↓
External DNS Name
  ↓
External Service
```

No kube-proxy involvement.

---

## Important Beginner Notes

* ExternalName is DNS-only
* No traffic inspection or routing by Kubernetes
* TLS and ports are handled by the client application
* Not suitable for protocols requiring IP-based routing

---

## Beginner Takeaway

* ExternalName = DNS alias inside Kubernetes
* Simplifies external service access
* Ideal for abstraction and migration
* Not a replacement for ClusterIP or LoadBalancer
