# 04.01 – The Foundation

# Kubernetes Service Types

In Kubernetes, a **Service** is an abstraction that **exposes a set of Pods** as a **network service**.
Because Pods are ephemeral (IP changes when recreated), Services provide **stable networking**.

---

## Why Services Are Needed

* Pods have **dynamic IPs**
* Pods can be **scaled up/down**
* Traffic must be **load-balanced**
* Applications need **stable DNS names**

A Service solves all of the above.

---

## Types of Kubernetes Services

Kubernetes supports **four primary Service types**:

1. **ClusterIP** (Default)
2. **NodePort**
3. **LoadBalancer**
4. **ExternalName**

---

## 1. ClusterIP (Default)

### What it does

* Exposes the Service **only inside the cluster**
* Not accessible from outside Kubernetes

### Use cases

* Internal microservice communication
* Backend APIs
* Databases

### Key points

* Gets a **virtual IP (VIP)** inside the cluster
* Accessible via **Service DNS name**

### Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: backend-service
spec:
  type: ClusterIP
  selector:
    app: backend
  ports:
  - port: 80
    targetPort: 8080
```

### Access pattern

```
frontend-pod → backend-service → backend-pod
```

---

## 2. NodePort

### What it does

* Exposes the Service on **each Node’s IP**
* Uses a **static port** in range `30000–32767`

### Use cases

* Development environments
* Quick testing
* When no cloud load balancer is available

### Key points

* Accessible using:

  ```
  <NodeIP>:<NodePort>
  ```
* Internally still uses ClusterIP

### Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-nodeport
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 30080
```

### Access pattern

```
User → NodeIP:30080 → Service → Pod
```

---

## 3. LoadBalancer

### What it does

* Exposes the Service using a **cloud provider load balancer**
* Automatically provisions an **external IP**

### Use cases

* Production applications
* Public-facing services
* Cloud environments (AWS, GCP, Azure)

### Key points

* Works on top of NodePort
* Requires cloud support or MetalLB (on-prem)

### Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
```

### Access pattern

```
User → Cloud LoadBalancer → NodePort → Pod
```

---

## 4. ExternalName

### What it does

* Maps a Service to an **external DNS name**
* No selector, no Pods

### Use cases

* Connecting to external APIs
* Databases outside the cluster
* Legacy services

### Key points

* Works via **DNS CNAME**
* No proxying or load balancing by Kubernetes

### Example

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  type: ExternalName
  externalName: db.example.com
```

### Access pattern

```
app → external-db.default.svc.cluster.local → db.example.com
```

---

## Service Type Comparison Table

| Service Type | Scope        | External Access | Typical Use Case       |
| ------------ | ------------ | --------------- | ---------------------- |
| ClusterIP    | Internal     | No              | Internal microservices |
| NodePort     | Node-level   | Yes (manual)    | Dev / testing          |
| LoadBalancer | Public       | Yes (automatic) | Production apps        |
| ExternalName | External DNS | Yes (DNS only)  | External services      |

---

## Important Notes for DevOps Engineers

* **Ingress is NOT a Service type**

  * It works **on top of Services**
* Prefer:

  * `ClusterIP` for internal traffic
  * `Ingress + ClusterIP` for HTTP/HTTPS apps
* Avoid `NodePort` in production unless required

---

## Real-World Example Architecture

```
Internet
   ↓
Ingress Controller
   ↓
ClusterIP Service
   ↓
Application Pods
```
